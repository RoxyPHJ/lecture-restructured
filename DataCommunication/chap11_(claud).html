데이터링크 제어 (DLC)
Data Link Control

프레임 짜기 · 흐름/오류 제어 · PPP · 인증 프로토콜 — 데이터 링크층의 완전한 이해

01

데이터 링크층 구조

두 개의 서브레이어

데이터 링크층은 기능적으로 두 계층으로 나뉩니다.

위층 — DLC
Data Link Control — 흐름 제어, 오류 제어, 프레임 짜기를 담당합니다. 점대점 및 다중점 링크 모두 적용됩니다.
아래층 — MAC
Medium Access Control — 공유 매체에 대한 접근 제어를 담당합니다. 충돌 회피, 채널 할당 등이 이 계층에서 처리됩니다.
02

프레임 짜기 (Framing)

프레임의 두 가지 크기 유형
유형 특징 대표 사례
고정 크기 프레임 모든 프레임이 동일한 길이 → 길이 필드 불필요, 구현 단순 ATM 광역네트워크의 셀(Cell)
가변 크기 프레임 프레임 경계(시작/끝)를 명시적으로 표기해야 함
주로 LAN에서 사용		 이더넷, 문자·비트 중심 프로토콜
핵심 포인트
가변 크기 프레임에서는 "프레임이 끝나는 곳과 다음 프레임이 시작하는 곳"을 수신자가 알 수 있도록 프레임 경계를 명확히 지정해야 합니다. 이를 위해 플래그(Flag)를 사용합니다.
03

문자 중심 프로토콜 (Character-Oriented)

프레임 구조
  • ①
    8비트 문자 기반전달되는 데이터는 코딩 시스템(ASCII 등)의 8비트 문자 단위로 구성됩니다.
  • ②
    플래그(Flag) 추가프레임의 시작과 마지막에 특정 문자 패턴(플래그)을 추가하여 경계를 표시합니다.
  • ③
    문자 채워넣기(Stuffing)데이터 내에 플래그 패턴이 등장하면 오인식 방지를 위해 이스케이프 문자를 삽입합니다.
문자 채워넣기 (Character Stuffing)

데이터 내에 플래그 문자와 동일한 패턴이 포함되면, 수신자가 이를 프레임 경계로 오인할 수 있습니다. 이를 방지하기 위해:

Stuffing 규칙
데이터 내 플래그 패턴 앞에 이스케이프 문자(ESC)를 삽입(Stuffing)하여 전송합니다.
수신 측에서는 ESC를 제거(Unstuffing)하여 원래 데이터를 복원합니다.
만약 데이터 내에 ESC 자체가 있다면, ESC 앞에 또 다른 ESC를 삽입합니다.
04

비트 중심 프로토콜 (Bit-Oriented)

비트 중심 프로토콜 특징
  • ①
    비트 열 인식프레임의 데이터 부분을 문자 단위가 아닌 전체 비트 열(bit stream)로 인식합니다.
  • ②
    플래그 패턴01111110 (6개의 1이 연속) 비트 패턴을 플래그로 사용합니다.
  • ③
    비트 채워넣기(Bit Stuffing)5개의 연속된 1 다음에 0을 강제로 삽입합니다.
왜 5개 이후에 삽입하는가?
플래그는 01111110으로 6개의 연속된 1을 포함합니다. 데이터에서 5개의 연속 1이 등장하면 그 뒤에 0을 삽입함으로써, 수신자는 실제 플래그(6개의 1)와 일반 데이터를 구별할 수 있습니다.
05

흐름 제어 & 오류 제어

흐름 제어 (Flow Control)

송신자가 확인응답(ACK)을 받기 전에 보낼 수 있는 데이터의 양을 제한하기 위해 사용하는 일련의 절차입니다.

목적
수신자가 처리할 수 있는 속도보다 빠르게 데이터가 도착하지 않도록 조절합니다.
오류 제어 (Error Control)

ARQ(Automatic Repeat reQuest)를 기반으로 하여, 손상되거나 분실된 프레임의 재전송을 요구합니다.

ARQ 방식
수신자가 오류를 감지하면 송신자에게 재전송 요청(NAK) 또는 무응답을 통해 재전송을 유발합니다.
06

DLC 프로토콜 종류

프로토콜 흐름제어 오류제어 특징
단순(Simple) 프로토콜 없음 없음 매체가 완벽하다고 가정, 단방향 전송
정지/대기(Stop-and-Wait) 프로토콜 있음 있음 한 번에 한 프레임 전송 후 ACK 대기
Go-Back-N 프로토콜 슬라이딩 윈도우 있음 오류 발생 시 해당 프레임부터 모두 재전송
선택적-반복(Selective-Repeat) 프로토콜 슬라이딩 윈도우 있음 오류가 발생한 프레임만 선택적으로 재전송
정지/대기 프로토콜 동작
  • 1
    프레임 전송송신자가 프레임 0을 전송합니다.
  • 2
    ACK 대기ACK가 올 때까지 송신자는 다음 프레임을 보내지 않습니다.
  • 3
    ACK 수신 후 전송ACK 수신 확인 후 다음 프레임을 전송합니다.
  • 4
    타임아웃 재전송일정 시간 내 ACK 없으면 해당 프레임을 재전송합니다.
단점
ACK를 기다리는 동안 링크가 유휴 상태 → 효율이 낮음
07

PPP (Point-to-Point Protocol)

PPP 개요

PPP는 점대점(Point-to-Point) 액세스에 사용되는 문자 중심 프로토콜입니다.

사용 환경
PC와 워크스테이션 간 모뎀/케이블 연결
적용 분야
전화회선을 통한 서버 연결 (인터넷 접속)
통신 방식
직렬 인터페이스를 이용한 데이터 송수신
PPP 프레임 형식 (Figure 11.32)
  • F
    Flag (1바이트)01111110 — 프레임 시작/끝
  • A
    Address (1바이트)11111111 — 브로드캐스트 주소
  • C
    Control (1바이트)00000011 — 비번호 정보
  • P
    Protocol (2바이트)페이로드의 프로토콜 유형 식별
  • D
    Data (가변)실제 전송 데이터 (페이로드)
  • E
    FCS (2 or 4바이트)Frame Check Sequence — 오류 검출

PPP는 여러 상위 계층 프로토콜(IP, IPX 등)을 단일 링크를 통해 다중화할 수 있습니다. Protocol 필드로 어떤 상위 프로토콜의 데이터인지 식별합니다.

08

PPP 인증 프로토콜

항목 PAP CHAP
전체 명칭 Password Authentication Protocol Challenge Handshake Authentication Protocol
단계 수 2단계 3단계 (3-way handshake)
보안 수준 낮음 (평문 전송) 높음 (챌린지-응답 방식)
패스워드 노출 직접 전송 → 도청 위험 직접 전송하지 않음 → 안전
재인증 연결 수립 시 1회 연결 중 주기적 재인증 가능
PAP 동작 (Figure 11.36)
  • 1
    자격증명 전송사용자가 사용자이름(Username)과 패스워드(Password)를 시스템에 전송합니다.
  • 2
    수용 또는 거부시스템이 자격증명을 확인하고 연결을 수용(ACK) 또는 거부(NAK)합니다.
보안 취약점
패스워드가 평문(plaintext)으로 전송되어 도청에 취약합니다.
CHAP 동작 (Figure 11.37)
  • 1
    챌린지 전송시스템이 난수를 포함한 챌린지 패킷을 사용자에게 전송합니다.
  • 2
    응답 전송사용자는 미리 정의된 함수(Hash)를 챌린지에 적용하여 결과를 시스템에 전송합니다.
  • 3
    검증 및 허용/거부시스템도 동일한 함수를 적용하여 결과를 비교합니다. 일치하면 허용, 불일치하면 거부합니다.
보안 강점
패스워드가 직접 전송되지 않고, 매번 다른 챌린지를 사용하여 재생(Replay) 공격에도 강합니다.
09

핵심 정리 요약

🏗️

DLC / MAC 구조

데이터링크층은 DLC(흐름·오류제어)와 MAC(매체접근) 두 서브레이어로 분리됩니다.

📦

프레임 짜기

고정 크기(ATM)와 가변 크기(LAN) 두 방식으로 나뉘며, 가변 크기는 플래그로 경계를 지정합니다.

🔤

문자 중심 프로토콜

8비트 문자 기반, ESC 삽입(Character Stuffing)으로 플래그 오인 방지.

💡

비트 중심 프로토콜

플래그 01111110, 5개 연속 1 뒤 0 삽입(Bit Stuffing).

🔄

흐름/오류 제어

흐름제어: ACK 전까지 전송량 제한. 오류제어: ARQ 기반 재전송 요구.

📡

DLC 프로토콜 4종

단순 → 정지/대기 → Go-Back-N → 선택적-반복 순으로 효율 증가.

🔗

PPP

점대점 문자중심 프로토콜. 5단계(Dead→Establish→Auth→Network→Open→Terminate) 전환.

🔐

PAP vs CHAP

PAP: 2단계, 평문 전송, 보안 취약. CHAP: 3단계, 챌린지-응답, 패스워드 미전송.
